> 版本：v1 — 阶段 0 规则冻结产物
> 适用项目：`asset_ledger_app`
> 文档地位：本文件是后续阶段 A/B/C/D/E 一切实现与测试的"权威规则源"。后续代码与测试出现歧义时，以本文件为准；如需调整规则，必须先在此处更新版本号并留下变更记录。

## 0. 通用约定

- 货币单位在内部统一使用 **fen（分）整数**，参见第 3 章。
- "结清（settled）" = 项目财务已确认终结，不再被自动复用。
- "active 项目" = 尚未结清的项目。
- "legacy_project_key" = 旧版用于复用项目身份的键，本阶段保留但其唯一性约束需在阶段 A 中按本规则修复。
- "外协" = external_work_record（外部协作记录）。
- "驾驶员" = 计时执行端，由老板审核后才生成正式 TimingRecord。
- "合伙人" = 通过分享设备方式获得只读访问能力的协作者。

## 1. 项目身份规则（Project Identity）

**核心结论：同联系人 + 同工地 + 已结清后再次开工 = 创建新项目。**

具体规则：

1. 已结清项目不再被自动复用。
2. 同联系人 + 同工地再次记工时，如果旧项目已结清，必须创建新的 `project_id`。
3. 旧项目保持结清历史，不被改写。
4. 未结清时，同联系人 + 同工地继续记工，自动归入当前 active 项目。
5. 同一个 `legacy_project_key` 下允许多个历史已结清项目共存。
6. 同一个 `legacy_project_key` 下**只允许一个未结清 active 项目**。

后续阶段（阶段 A）依据本规则修复 `legacy_project_key` 的唯一约束：从"全局唯一"调整为"同 key 下仅一个 active"。

## 2. 备份恢复规则（Backup / Restore）

**当前阶段只支持"覆盖恢复"，不支持"合并恢复"。**

恢复过程：

1. 恢复前清理当前本地业务数据。
2. 恢复包里的数据作为新的完整本地状态。
3. 恢复过程必须事务化。
4. 任何不可恢复的结构性错误必须失败并回滚。

外协缺失关联项目的特例：

- 如果 `external_work_record.linked_project_id` 在恢复包中找不到对应项目，**外协记录保留，但自动解除关联**。
- 保留 `external_work_record` 本体。
- `linked_project_id` 置空。
- `linked_at` / linked 状态清空。
- `RestoreResult` 必须记录 warning。
- UI 后续应提示："部分外协记录已恢复为未关联状态"。
- 不允许静默处理。

## 3. 金额规则（Money）

**内部统一按 fen 存储和计算。**

具体规则：

1. 数据库、计算、汇总统一使用 `amount_fen`（整数分）。
2. 从旧 `amount`（REAL 元）迁移时：`amount_fen = round(amount * 100)`。
3. UI 展示时可以转换为元（除以 100），可按需保留 2 位小数或取整展示。
4. 财务判断、结清判断、核销判断、汇总判断**不得依赖** REAL 元字段。
5. `amount`（REAL 元）仅作为 legacy / display 兼容字段，不能作为权威计算来源。

注意事项：

- 不要把内部金额直接四舍五入到元。
- 整数分不会比元更低效，且能避免 REAL 小数精度误差。
- 旧 `amount` 字段允许在 UI/导出层读取以兼容历史展示，但任何"是否结清 / 是否核销 / 是否欠款"判断都必须基于 `amount_fen`。

## 4. 退款 / 冲正 / 调账规则（Refund / Reversal / Adjustment）

**后续会支持，但当前阶段不立即实现。**

当前阶段约束：

1. 普通收款不允许用负数表达退款。
2. 普通核销不允许用负数表达冲正。
3. 普通外协金额不允许为负数。
4. 后续应通过独立类型表达：
   - refund（退款）
   - adjustment（调账）
   - reversal（冲正）
5. 退款 / 冲正 / 调账必须保留原因和审计信息（actor、reason、original_ref、timestamp）。

---

## 5. 0 元项目结清规则（Zero-amount Settlement）

**没有任何应收、实收、核销、外协费用的项目，不能结清。**

具体规则：

1. 项目仅在新建状态、没有任何业务金额时，**不能结清**。
2. 当前阶段不支持"免费项目结清"。
3. 后续如果需要支持免费帮忙 / 人情活，应单独设计：
   - `zero_amount_settlement_reason`
   - 或 `free_settlement_type`

---

## 6. 修改计时导致项目变化的规则（Timing Edit → Project Re-assignment）

**修改计时记录导致项目变化时，系统自动解除受影响的合并关系。**

具体规则：

1. 保存计时记录与解除合并必须在同一个事务中完成。
2. 任一步失败，整体回滚。
3. 不允许出现"计时已保存，但合并未解除"的中间状态。
4. 不再依赖 UI 内存中的 pending retry 队列作为一致性保障。
5. 保存成功后 UI 可以提示："已自动解除合并 / 已自动撤销结清"。

---

## 7. 合并解除后的结清规则（Post-Unmerge Settlement Recompute）

**合并解除后，系统允许自动撤销不再成立的结清状态。**

具体规则：

1. 不自动删除收款。
2. 不自动删除核销。
3. 不自动删除原始计时。
4. 重新计算每个受影响项目的应收、已收、核销、未收。
5. 如果拆分后原结清状态不再成立，**自动撤销结清**。
6. UI 必须给出明确提示（不能静默撤销）。
7. 本逻辑应与"删除计时影响结清"的规则保持一致，复用同一套 settlement impact 计算。

## 8. 驾驶员记录规则（Driver Review）

**驾驶员记录审核后生效。**

状态机：

| 状态 | 说明 |
|------|------|
| `draft` | 驾驶员可改 |
| `submitted` | 驾驶员确认上传后不可改 |
| `approved` | 老板审核通过后生成或更新正式 TimingRecord |
| `rejected` | 老板拒绝，必须有拒绝原因 |

权限：

1. 驾驶员可修改 `draft`。
2. 驾驶员不可修改 `submitted`。
3. 老板可审核、拒绝、修改正式记录。
4. 老板修改后必须保留 audit 记录。
5. 建议保留驾驶员的原始提交内容；正式 TimingRecord 保存老板审核后的内容（即两份共存，便于追溯）。

## 9. 合伙人共享规则（Partner Sharing）

**合伙人同步只允许一端写入，另一端只读。**

具体规则：

1. 分享设备的双方都应可读查看。
2. 未分享设备不可查看。
3. 一端为写入端，另一端为只读端。
4. 当前阶段不做双向同时编辑。
5. 合伙人可以看被分享设备的计时数据。
6. 合伙人不能看其它设备。
7. 默认不开放：
   - 老板总利润
   - 全项目总账
   - 未分享设备收入
8. 由于当前 Excel 仅导出简单计时数据（见第 10 章），合伙人第一版不涉及复杂财务权限。

## 10. Excel / 导出规则（Simple Device Work-Hours Export）

**Excel 当前只做简单设备工时表，不做复杂财务报表。**

第一版导出列：

- 设备
- 计时日期
- 开始码表
- 结束码表
- 码表差
- 开始时间
- 结束时间
- 工时
- 直接输入工时
- 联系人
- 工地
- 备注

**不包含**：

- 应收
- 已收
- 核销
- 未收
- 利润
- 外协成本
- 项目总账
- 老板利润
- 合伙人分成

后续如需做 Excel，**只需要设备工时表**，不需要复杂 `ProjectFinanceReportViewModel`。

## 11. AI / MCP 规则（AI / MCP Preview-Confirm）

**AI / MCP 必须只能预览，用户确认后才能执行。**

统一流程：

```
parse -> disambiguate -> preview -> confirm -> execute -> audit
```

高风险操作（必须走完整流程）：

- 删除
- 结清
- 核销
- 恢复备份
- 批量导入
- 批量修改项目
- 批量外协关联
- 修改已审核记录

规则：

1. AI / MCP 不允许直接改账。
2. 第一版适合只做：
   - 只读查询
   - 生成计时草稿
   - 生成导出预览
   - 生成外协关联预览
   - 生成删除影响预览
3. 所有写操作必须走 `preview + confirm + audit`。

---

## 12. 删除 / 同步基础规则（Soft Delete & Sync Foundations）

**核心业务删除先软删除。**

具体规则：

1. 使用 `deleted_at` / tombstone 表达软删除。
2. 同步层保留删除事件。
3. UI 默认隐藏 `deleted_at` 不为空的数据。
4. 核心业务表后续需要逐步补齐：
   - `created_at`
   - `updated_at`
   - `deleted_at`
   - `owner_id`
   - `actor_id`
   - `version`
5. 核心业务表包括但不限于：
   - `timing_records`
   - `payments`
   - `write_offs`
   - `projects`
   - `external_work_records`
   - merge group 相关表
