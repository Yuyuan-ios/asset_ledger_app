# 机账通本地恢复策略设计

## 1. 目标

本地恢复功能的目标是允许用户从机账通导出的本地 JSON 备份中恢复账本数据。

- 允许用户从机账通导出的本地 JSON 备份中恢复账本数据。
- 第一版优先保证安全、可理解、可回滚。
- 恢复功能服务于数据安全，不追求复杂合并。
- 当前已完成预检与摘要预览，下一阶段才实现真实写入。

本设计文档只定义真实恢复策略，不实现恢复写入、不修改数据库 schema、不改变现有备份导出格式。

## 2. 第一版恢复模式

第一版只支持：完整覆盖恢复。

完整覆盖恢复的含义：

- 用户选择一个有效备份文件。
- 当前本机业务数据会被备份文件中的数据替换。
- 不支持合并恢复。
- 不支持追加恢复。
- 不支持部分表恢复。
- 不支持多备份合并。

第一版不做合并恢复，原因如下：

- 当前没有云端账号体系。
- 当前没有全局唯一 ID。
- 当前没有冲突解决机制。
- 合并恢复会带来重复记录、ID 冲突、项目 key 冲突、设备关系错乱等风险。
- 第一版以安全和可预测为优先。

因此，真实恢复入口必须明确告诉用户：恢复会替换当前本机业务数据，而不是把备份数据追加进去。

## 3. 恢复范围

第一版只恢复当前备份中的 6 张业务表：

- `devices`
- `timing_records`
- `fuel_logs`
- `maintenance_records`
- `account_payments`
- `project_device_rates`

第一版明确不恢复：

- 订阅状态
- Pro 权限
- App 临时缓存
- App UI 设置
- 本地图片文件
- 系统权限状态
- 第三方账号状态
- 未来云端登录状态

订阅状态必须来自 Apple / 服务端校验，不能从备份文件恢复。备份文件只承载结构化账本业务数据，不应成为权限、订阅或身份状态来源。

## 4. 恢复前自动备份规则

真实恢复前必须自动导出当前本机数据备份。

流程：

1. 用户选择备份文件。
2. 预检通过。
3. 展示备份摘要和风险提示。
4. 用户确认恢复。
5. App 自动导出当前本机数据备份。
6. 自动备份成功后，才允许进入数据库恢复事务。
7. 如果自动备份失败，终止恢复，不修改当前数据。

自动备份的意义：

- 给用户留退路。
- 防止误选旧备份造成不可逆损失。
- 降低恢复失败后的数据风险。

自动备份文件应沿用现有 `LocalBackupExportService.exportJsonBackup()` 能力。若自动备份失败，恢复流程必须立即停止，并给出清晰提示。

## 5. 数据库事务与回滚策略

真实恢复必须在数据库事务中执行。

要求：

- 清空旧业务表和写入备份数据必须放在同一个事务中。
- 任何一张表写入失败，整个事务回滚。
- 事务失败后，本机数据应保持恢复前状态。
- 不允许出现“部分表恢复成功、部分表失败”的状态。
- 恢复过程中 UI 应阻止重复点击。
- 恢复过程中应显示 loading 状态。

建议流程：

1. 开启事务。
2. 按外键/业务依赖顺序清空 6 张表。
3. 按依赖顺序写入 6 张表。
4. 提交事务。
5. 刷新相关 Store / 页面状态。
6. 展示恢复成功提示。

建议清空顺序：

1. `project_device_rates`
2. `account_payments`
3. `maintenance_records`
4. `fuel_logs`
5. `timing_records`
6. `devices`

建议写入顺序：

1. `devices`
2. `timing_records`
3. `fuel_logs`
4. `maintenance_records`
5. `account_payments`
6. `project_device_rates`

这样可以优先恢复设备，再恢复依赖设备或项目 key 的业务记录。

## 6. schemaVersion 兼容策略

第一版只允许 `meta.schema_version == AppDatabase.schemaVersion` 时执行真实恢复。

规则：

- schemaVersion 相同：允许恢复。
- schemaVersion 高于当前 App：拒绝恢复，提示用户升级 App。
- schemaVersion 低于当前 App：允许预览，但第一版不执行真实恢复。
- schemaVersion 缺失：判定为无效备份文件。
- export_format_version 缺失或不支持：判定为无效备份文件。

原因：

- 当前还没有旧版本备份迁移适配器。
- 低版本备份直接写入可能缺字段或字段含义不同。
- 高版本备份可能包含当前 App 不认识的数据结构。
- 先支持同版本恢复，后续再补版本迁移。

当前 `AppDatabase.schemaVersion` 为 `9`。第一版真实恢复只应接受 `meta.schema_version == 9` 的备份文件。

## 7. 自定义头像处理策略

当前 `devices` 中可能存在 `custom_avatar_path`。

第一版策略：

- 只恢复数据库字段。
- 不保证恢复本地图片文件。
- 如果 `custom_avatar_path` 指向的文件不存在，UI 应兜底显示默认头像。
- 不因为头像文件缺失而阻止账本数据恢复。
- 后续若要完整恢复头像，应设计“备份包”格式，而不是单纯 JSON。

说明：

JSON 备份只适合恢复结构化账本数据，不适合直接恢复本机图片文件。自定义头像文件若要被完整迁移，应在未来引入包含 JSON 与资源文件的备份包，例如 `.zip` 格式。

## 8. 数据校验规则

真实恢复前必须复用或扩展第 10 步的预检能力。

必须校验：

- 顶层必须是 Map。
- 必须包含 `meta`。
- 必须包含 `data`。
- `meta.app_name` 应符合机账通备份标识。
- `meta.schema_version` 必须存在。
- `meta.export_format_version` 必须存在。
- `data` 下 6 张业务表必须存在。
- 6 张业务表必须都是 List。
- 每张表中的记录必须是 Map。
- 必要字段缺失时应拒绝恢复或给出明确错误。
- 数字字段、日期字段、布尔字段应进行基础类型校验。

错误提示应用户可理解，例如：

- 这不是有效的机账通备份文件。
- 备份文件格式不完整。
- 备份文件版本较新，请升级 App 后再试。
- 当前版本暂不支持恢复旧版备份。
- 备份数据结构异常，无法恢复。

第一版真实恢复不应只检查列表数量，还应逐行检查字段类型。只有所有记录都通过校验，才允许进入事务。

## 9. 恢复确认文案

真实恢复前必须展示二次确认。

要求表达清楚：

- 当前本机业务数据将被备份文件替换。
- 恢复前会自动备份当前数据。
- 恢复过程中请勿关闭 App。
- 当前只支持完整覆盖恢复，不支持合并。
- 订阅状态和 App 设置不会从备份恢复。

建议文案：

标题：

确认恢复备份？

正文：

恢复后，当前本机的设备、计时、油费、维修、收款和项目相关设置等业务数据将被所选备份替换。恢复前，App 会先自动导出一份当前数据备份，便于必要时找回。当前版本仅支持完整覆盖恢复，不支持合并恢复。

按钮：

- 取消
- 确认恢复

这里不需要描述产品付费边界，只描述恢复影响范围。

## 10. 成功与失败反馈

恢复成功：

- 提示“恢复完成”。
- 刷新设备、计时、油费、维修、账户等相关状态。
- 返回账户中心或数据安全页面。
- 可提示用户检查数据是否正确。

恢复失败：

- 提示失败原因。
- 明确说明当前数据未被修改或已回滚。
- 如果自动备份已成功，应提示自动备份文件位置。
- 不要让用户误以为恢复已经部分成功。

失败提示建议：

- 恢复失败，当前数据未被修改。
- 恢复失败，数据库已回滚，当前数据保持恢复前状态。
- 恢复前自动备份失败，已终止恢复，当前数据未被修改。

## 11. 未来合并恢复策略

合并恢复不在第一版实现。

未来如需支持，需要先设计：

- 全局唯一业务 ID
- 旧 ID 到新 ID 的映射表
- 设备去重规则
- 项目 key 冲突规则
- 计时记录重复判断
- 收款记录重复判断
- 关联关系重建
- 合并前预览差异
- 合并后撤销策略

结论：

没有这些基础前，不应实现合并恢复。

## 12. 与云端备份的关系

本地恢复策略会成为未来云端恢复的基础。

第一阶段云端能力建议是：云端备份包 + 换机恢复，而不是实时多端同步。

说明：

- 本地 JSON 备份结构可作为云端备份包的基础。
- 云端恢复也应先走预检和摘要预览。
- 云端恢复第一版也应优先考虑覆盖恢复。
- 实时多端同步涉及冲突解决、离线写入、删除同步、版本合并，暂不适合当前阶段。

未来云端备份可以复用以下流程：

1. 拉取云端备份元信息。
2. 展示备份摘要。
3. 校验 App 与 schemaVersion 兼容性。
4. 自动备份当前本机数据。
5. 在事务中执行覆盖恢复。
6. 刷新本机状态。

## 13. 第一版真实恢复 MVP 验收标准

第一版真实恢复 MVP 需要满足以下验收标准：

- 只能从机账通 JSON 备份文件恢复。
- 只允许恢复 `meta.schema_version == AppDatabase.schemaVersion` 的备份。
- 恢复前必须展示备份摘要。
- 恢复前必须展示二次确认。
- 恢复前必须自动导出当前本机备份。
- 自动备份失败时必须终止恢复。
- 恢复写入必须在单个数据库事务中完成。
- 事务失败后当前数据必须保持恢复前状态。
- 恢复成功后必须刷新相关页面数据。
- 恢复成功后必须给出明确成功提示。
- 恢复失败后必须给出明确失败原因。
- 不恢复订阅状态、Pro 权限、App 设置、本地图片文件。
- 不支持合并恢复、追加恢复、部分表恢复。

## 14. 第一版实现边界

第一版真实恢复可以新增：

- 本地恢复执行服务。
- 更严格的备份记录级校验。
- 覆盖恢复确认弹窗。
- 恢复 loading 状态。
- 恢复成功后的 Store 刷新流程。

第一版真实恢复不应新增：

- 数据库 schema 变更。
- 云端备份或云端恢复。
- 多端实时同步。
- 合并恢复。
- 订阅权限恢复。
- Pro 限制。
- 图片文件打包恢复。

本地备份和本地恢复保持当前产品定位，作为数据安全链路的一部分。本策略不引入新的订阅限制。

## 15. 推荐实施顺序

建议按以下顺序实现真实恢复：

1. 扩展预检服务，增加逐行字段校验。
2. 新增恢复确认弹窗，只在 schemaVersion 完全一致时显示“确认恢复”。
3. 新增恢复执行服务，但先只封装事务流程，不接 UI。
4. 接入恢复前自动备份，自动备份失败则终止。
5. 在事务中实现 6 张业务表的覆盖写入。
6. 恢复成功后刷新 DeviceStore 及相关页面状态。
7. 补充同版本备份恢复的手工验收清单。

每一步都应保持“可预览、可取消、可回滚、可解释”。

## 16. 测试用例清单

### 有效场景

- 当前版本导出的完整备份可以预览。
- schemaVersion 相同的备份可以进入恢复确认。
- 恢复前自动备份成功后才开始写库。
- 恢复成功后 6 张业务表数量与备份摘要一致。

### 异常场景

- 用户取消选择文件。
- 非 JSON 文件。
- JSON 解析失败。
- 顶层不是 Map。
- 缺少 meta。
- 缺少 data。
- 缺少某张业务表。
- 某张业务表不是 List。
- schemaVersion 缺失。
- schemaVersion 高于当前版本。
- schemaVersion 低于当前版本。
- 自动备份失败。
- 写入中途失败。
- 用户重复点击恢复按钮。
- 恢复过程中 App 被切到后台或中断。

### 边界场景

- 空备份。
- 只有设备，没有记录。
- 大量计时记录。
- 自定义头像路径不存在。
- project_device_rates 存在但关联设备不存在。
- 备份中包含未知额外字段。
