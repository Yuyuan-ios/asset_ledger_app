# 问题标题
2026-03-01，Store 职责过重与依赖方向收口

## 问题描述
多个 Store 同时承担状态、CRUD 编排、筛选与聚合逻辑，且直接依赖静态 Repo，导致边界不清晰，后续测试与替换实现成本较高。

## 根本原因分析
- 早期以快速上线为目标，采用静态仓库与轻量 Store，降低了接入成本，但逐步累积了职责混杂。
- 聚合编排逻辑直接放在 Store 中，使状态层和应用层边界模糊。

## 解决方案与步骤
1. 将计时、燃油、设备、维保、收款、项目单价覆盖等核心仓库改为“接口 + sqflite 实现”。
2. 将各 Store 改为构造注入具体 Repository，并在 `AppProviders` 中统一装配。
3. 将账户聚合编排从 `AccountStore` 下沉到 `ComputeAccountSummaryUseCase`，保留页面调用方式不变。
4. 将计时与燃油的联想逻辑拆到独立 service，避免 Store 继续膨胀。

## 改进启示
- 先保持行为不变，再调整依赖方向，是低风险重构的有效路径。
- Store 应优先承担状态协调，复杂编排应进入 use case / service，才能维持长期可维护性。
